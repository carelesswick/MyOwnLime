# FFT 性能优化案例：从 8.2s 到 0.43s（19 倍加速）

## 一句话总结

通过 `cv::getOptimalDFTSize()` 将图像尺寸对齐到 FFT 友好尺寸，在一个函数内部用 padding/crop 策略，将 LIME 算法的 FFT 耗时从 8.2s 降至 0.43s，**对外部逻辑零影响**。

---

## 1. 问题发现

| 测试图像 | 尺寸 | 质因数分解 | ADMM 总耗时 |
|---------|------|-----------|------------|
| 图像 A | 1039 × 789 | 1039 = **质数**, 789 = 3 × **263** | **8.2 s** |
| 图像 B | 1024 × 768 | 1024 = **2¹⁰**, 768 = 3 × **2⁸** | **0.395 s** |

同一套代码，图像 B 比图像 A 快 **20.8 倍**。两张图像素数量相当（82 万 vs 79 万），不可能是运算量差异。

---

## 2. 根因分析

### 2.1 FFT 的时间复杂度不是只看 N

FFT 的理论复杂度是 O(N log N)，但**常数因子**取决于 N 的质因数分解：

| N 的类型 | 示例 | 算法 | 相对速度 |
|----------|------|------|---------|
| 2 的幂 | 1024 = 2¹⁰ | Radix-2 Cooley-Tukey | 1×（最快） |
| 小质因子乘积 | 1080 = 2³×3³×5 | 混合基 FFT | ~2-5× |
| 含大质因子 | 789 = 3×263 | Bluestein / 质因子算法 | ~10-30× |
| **质数** | **1039** | **Chirp-Z / Bluestein** | **~20-50×** |

### 2.2 本案例的 FFT 调用量

ADMM 每轮迭代执行 **2 次 FFT**（1 次正变换 + 1 次逆变换），10 轮 = **20 次 FFT**。

对于 1039×789：
- 单次 FFT 耗时约 0.4s
- 20 次 FFT = **8s**

对于 1024×768：
- 单次 FFT 耗时约 0.02s
- 20 次 FFT = **0.4s**

### 2.3 质因数可视化

```
图像 A:  1039 = 1039              ← 质数本数，FFT 最坏情况
          789 = 3 × 263           ← 263 是质数

图像 B:  1024 = 2 × 2 × 2 × 2 × 2 × 2 × 2 × 2 × 2 × 2  ← 纯 2 的幂
          768 = 2 × 2 × 2 × 2 × 2 × 2 × 2 × 2 × 3       ← 2⁸×3
```

---

## 3. 解决方案

### 3.1 核心思路

不改外部接口，只在 `SolveT` 内部：

1. `cv::getOptimalDFTSize()` 查询当前尺寸的最优 FFT 尺寸
2. 如果需要 padding：把 `rhs` 和 `kernel` 扩到最优尺寸做 FFT
3. 逆变换后 crop 回原始尺寸

对外部 ADMM 完全透明——T、G、Λ、T_hat 的所有矩阵仍保持原始尺寸。

### 3.2 关键代码

```cpp
// 查询 OpenCV 内部的最优 FFT 尺寸
const int optH = cv::getOptimalDFTSize(H);   // 789  → 800
const int optW = cv::getOptimalDFTSize(W);   // 1039 → 1080
const bool need_pad = (optH != H || optW != W);

if (need_pad) {
    // 反射填充（近似 Neumann 边界条件，优于零填充）
    cv::copyMakeBorder(rhs, rhs_padded,
        0, optH - H, 0, optW - W, cv::BORDER_REFLECT);

    // 用 padded 尺寸生成 kernel 并做 FFT
    cv::Mat kernel = GenerateLaplacianFreqKernel(pad_sz, rho);
    cv::dft(rhs_padded, s_fft, cv::DFT_COMPLEX_OUTPUT);

    // ... 频域乘法 ...

    cv::idft(s_fft, T_padded, cv::DFT_SCALE | cv::DFT_REAL_OUTPUT);

    // 裁剪回原始尺寸
    T = T_padded(cv::Rect(0, 0, W, H)).clone();
}
```

### 3.3 为什么 BORDER_REFLECT 而不是零填充

PDE 的边界条件是 Neumann（零梯度）。`BORDER_REFLECT` 在边界处镜像反射，保持梯度连续性，比零填充（会引入阶跃）更接近真实边界条件，减少边界处的 FFT 振铃效应。

## 5. 延伸知识点（面试可能会问）

### 5.1 `getOptimalDFTSize` 的原理

OpenCV 内部维护一张"好尺寸"表，只包含质因子为 2、3、5 的数字。它返回 ≥N 的最小"好尺寸"：

```
getOptimalDFTSize(1039) = 1080 = 2³ × 3³ × 5
getOptimalDFTSize(789)  = 800  = 2⁵ × 5²
```

这些尺寸可以用混合基 FFT 高效计算。

### 5.2 为什么不全用 2 的幂

2 的幂确实最快，但 padding 过度会增加计算量：
- 1039 → 2048：面积翻 4 倍，FFT 不一定更快
- 1039 → 1080：面积增 7%，FFT 提速 20×，最优折中

`getOptimalDFTSize` 自动做了这个权衡。

### 5.3 这个优化思路的通用性

任何用到 FFT 的算法（卷积、去模糊、频域滤波、相位相关）如果输入尺寸含大质因子，都可以用同样的 padding 策略。典型场景：

- 图像去模糊（Wiener 滤波）
- 相位相关图像配准
- 频率域水印
- 医学图像重建

## 7. 关键教训

> **"FFT 慢，不是代码写得不好，是输入尺寸不好。"**

当遇到 FFT 性能问题时：
1. 先检查输入尺寸的质因数分解
2. 用 `getOptimalDFTSize` 找最近的好尺寸
3. 内部 padding/crop，外部零感知
4. 不要自己实现 FFT——算法层面比实现层面重要 100 倍
